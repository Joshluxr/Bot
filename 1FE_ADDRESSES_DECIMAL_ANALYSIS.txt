All hexadecimal keys show the SAME pattern:

  [VARIED PREFIX] + [MANY ZEROS] + [SMALL OFFSET]

Example Hex Keys:
  0x637d18efcf30e1aea1452d0595c73aa299916abe7f1ec8000000000005936846
     └─────────────── Prefix ────────────┘└──── Zeros ────┘└─ Offset ─┘

  0x425365f534cb411f162e1e03b92f7c6c6660f1d454bf30000000000001832f99
     └─────────────── Prefix ────────────┘└──── Zeros ────┘└─ Offset ─┘

Pattern: ~48 hex digits prefix + ~10 zeros + ~8 hex digit offset

Why This Creates Vanity Prefixes:
- The decimal base (10, 20, 30...80 × 10^76) creates predictable hex prefixes
- These hex prefixes translate to Bitcoin addresses via ECDSA
- Result: "Accidental" vanity addresses like 1Fe, 1A1, 12c, etc.

1. SYSTEMATIC GENERATION
   - Not random keyspace exploration
   - Follows decimal base × 10^72 pattern
   - Predictable intervals (5 or 10 × 10^76)

2. VANITY-LIKE PREFIXES ARE ACCIDENTAL
   - Not traditional vanity mining (brute force)
   - Result of systematic decimal structure
   - Creating hex keys with many trailing zeros

3. KEYSPACE LOCATION
   - All keys in range: 10^76 to 80^76
   - This is a TINY fraction of Bitcoin's 2^256 keyspace
   - 2^256 ≈ 1.16 × 10^77 (much larger than 10^76)

4. WHY TRAILING ZEROS?
   - Decimal values like 45 × 10^76 have simple structure
   - Converting to hex creates patterns with many zeros
   - Trailing zeros don't affect address generation

5. SECURITY STATUS
   - ✅ All 92 addresses have ZERO balance
   - ✅ No funded addresses discovered
   - ✅ Demonstrates Bitcoin's cryptographic strength
   - ✅ Even systematic exploration yields no results
